### Вывод по фактической базе

- В 1С реально используется большой контур отгрузок: 19 552 расходные накладные и 90 944 строки. В текущем приложении отгрузки ранее были пустыми, значит импорт/общая БД не покрывают ключевой сценарий.
- Закупки и приемка в 1С не тестовые: 588 заказов поставщику, 1 844 приходные накладные, 842 строки расходов в приходах и 86 документов доп. расходов.
- Складской контур в 1С живой: 911 перемещений запасов, 193 инвентаризации, 162 списания, 158 оприходований.
- Цены в 1С ведутся документами установки цен: 683 документа и 99 026 строк.
- Деньги в 1С являются отдельным рабочим контуром: касса, банк, расходы, кассовые смены.
- В 1С 1 051 роль. Повторять их один-в-один нельзя, но нужно сделать маппинг рабочих ролей MajorWarehause на реальные группы прав.

## Что уже есть в MajorWarehause

### Общая БД и режим клиента

- Конфигурация читается из `appsettings.json`, `appsettings.local.json` и переменных окружения.
- В репозиторном `WarehouseAutomatisaion.Desktop.Wpf/appsettings.json` сейчас `RemoteDatabase.Enabled = false`.
- `SalesWorkspaceStore` поддерживает серверный режим: если remote включен и сохранить в сервер не удалось, локальное сохранение отключается ошибкой.
- `MainWindow` запускает периодический refresh только для sales workspace.
- `CatalogWorkspaceStore`, `PurchasingOperationalWorkspaceStore` и `WarehouseOperationalWorkspaceStore` умеют читать/писать backplane snapshots, но при недоступном backplane сохраняют локальные JSON-файлы. Для строгой общей БД это надо закрыть отдельно.

### Рабочие разделы

| Раздел | Что закрывает сейчас |
|---|---|
| Главная | Рабочая панель, переходы, базовые показатели. |
| Заказы | Заказы покупателей, действия по подтверждению, резерву, счету, отгрузке, оплате и закрытию. |
| Клиенты | Список клиентов, карточка, статусы активности. |
| Счета | Счета клиентам, выдача, отметка оплаты, создание отгрузки. |
| Отгрузки | Список отгрузок, подготовка и отгрузка. |
| Закупки | Поставщики, заказы поставщику, счета поставщика, приходные накладные. |
| Склад | Перемещения, инвентаризация, списания, складские рабочие действия. |
| Товары | Номенклатура, виды цен, скидки, установка цен, журнал, выгрузка прайс-листа. |
| Отчеты | Аналитика по продажам, статусам и ключевым данным. |
| Связи данных | Карта сценариев и встроенный аудит целостности данных. |

### Документные цепочки

- Продажи: `Покупатель -> Заказ покупателя -> Счет -> Отгрузка -> Оплата/закрытие`.
- Закупки: `Поставщик -> Заказ поставщику -> Счет поставщика -> Приходная накладная`.
- Склад: `Перемещение`, `Инвентаризация`, `Списание`.
- Каталог: `Товар -> Вид цены -> Установка цены -> Скидка -> Прайс-лист`.

## Что нужно проверить в живой 1С

После первичного подключения к живой базе уже подтверждены ключевые документы и объемы. Дополнительно нужно проверить:

- фактический список разделов, документов, отчетов, обработок и доработок;
- обязательные поля в формах, которые реально заполняют сотрудники;
- роли пользователей и запреты по операциям;
- печатные формы, которые используются в работе;
- реальные статусы документов и правила проведения;
- кастомные реквизиты, расширения, внешние обработки;
- реальные связи между заказами, закупками, резервами, отгрузками и оплатами;
- проблемные места миграции по конкретным ссылкам, а не только по количествам.

## Предлагаемый первый участок реализации

Начать лучше не с отчетов и не с мелких кнопок, а с P0 по общей БД:

**Строгий общий режим БД для всех рабочих модулей.**

Почему это первый участок:

- уже был реальный симптом: один пользователь добавил заказ, другой его не увидел;
- пока есть тихий локальный fallback, нельзя доверять многопользовательским данным;
- 1С в рабочем режиме всегда дает общий источник правды, и MajorWarehause должен вести себя так же;
- после этого можно спокойно делать карточку заказа, резервы, роли и печать, не опасаясь расхождения данных.

Минимальный состав задачи:

1. Ввести единый признак строгого remote-mode для всех stores.
2. Отключить локальное сохранение для catalog/purchasing/warehouse, если remote включен.
3. Добавить понятный статус подключения к общей БД в главном окне.
4. Добавить диагностику: какой источник данных сейчас используется - сервер или локальный `app_data`.
5. Проверить сценарий двух клиентов: запись на одном рабочем месте видна на другом.

Следующим участком после этого должна идти **единая карточка заказа покупателя + цепочка документов**.
